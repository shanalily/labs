Task 4

  • I read Why Coding Style Matters by Nicholas C. Zakas
  • The link was dead, so I had to find the style guide for Java elsewhere. I haven't learned Java yet, so some parts meant little to me, but it helped show that companies like Google care about style. The rules don't seem too strict, and I'm guessing a lot of people use those rules anyway without thinking about it.
  • In the video, Douglas Crockford, who created JSLint, talks about programming style in Javascript. He thinks it is good to make coding style more strict, and be given warnings about style, so that code will be uniform and developers will have an easier time reading their code and debugging it. The prettiest or most concise code isn't always the clearest code. He compares it to writing, which has strict rules about punctuation. Some of the things he said made a lot of sense, like using "===" instead of "==" in Javascript, since that could easily be a bug someone doesn't pick up on. He also said to make global variables stick out by using caps, since they should be uncommon, which is something I've seen before. Probably the most controversial issue he mentioned was the use of "++". People like using it because it is shorter than writing "+= 1", and some may argue that it is prettier. Also x++ is not equal to x += 1, ++x is equal to x += 1. He says it is unnecessary to have another way to add 1, and he never uses "++". I personally like using it, as many other people do, so I plan to keep using it. However, I always use "++x" instead of "x++", which seems to be the better option. He also mentioned the switch statement issue of having cases fall through, which can be easy to do as an accident. I haven't used switch statements for any projects, I've only used them when learning languages, but I'll keep the problems of letting cases fall through in mind.
